# interior-instant 프로젝트 스터디

작성일: 2026-07-04
목적: 리뉴얼(디자인/구성 변경) 작업 전, 현재 구현된 껍데기 상태를 파악하기 위한 분석 문서.

## 1. 프로젝트 개요

- 참고 사이트: https://www.matchesinterior.com/ (섹션 흐름·기능을 참고용 베이스로 사용)
- 컨셉: "9.9% 마진 기준 원가 공개형 인테리어 상담·시공 랜딩페이지"
- 현재 상태: 실제 백엔드/CMS 없이 정적 HTML 2장 + 이미지 에셋으로만 구성된 프로토타입. README에도 "원가 공개 인테리어 랜딩 페이지 프로토타입"이라고 명시되어 있음.
- 배포: Vercel 정적 호스팅 전제 (`vercel.json`에 `cleanUrls: true`, `trailingSlash: false`만 설정).

## 2. 기술 스택

- 순수 HTML + 인라인 `<style>` + 인라인 `<script>`. 프레임워크, 빌드 도구, 패키지 매니저 전혀 없음 (`package.json` 없음).
- 폰트: Pretendard / Apple SD Gothic Neo / Noto Sans KR (웹폰트 링크 없이 시스템 폰트 폴백에 의존 — 실제 Pretendard 로드 안 됨).
- 이미지: `assets/images/`에 jpg/png/webp/jpeg 혼용, 배경 클래스(`img-*`)로 CSS에서 개별 매핑.
- 데이터: JS 객체 리터럴에 하드코딩 (DB/CMS/API 없음).

## 3. 파일 구조

```
interior-instant/
├── index.html          # 메인 랜딩 (1981줄, 전부 단일 파일)
├── portfolio.html       # 포트폴리오 상세 (1955줄, 전부 단일 파일)
├── vercel.json           # 정적 배포 설정
├── README.md             # 로컬 실행/배포/폼 연동 안내
└── assets/images/        # 히어로, 소장, 사례 이미지 (jpg/png/webp 혼용, 파일명 규칙 불일치)
```

git 히스토리(총 6커밋): 랜딩 초기화 → 콘텐츠/에셋 다듬기 → 포트폴리오 상세 페이지 연결 → 견적/문의 모달 추가 → 푸터 톤 정리. 즉 "링크만 살아있는 껍데기"를 순차적으로 붙여온 흐름.

## 4. 디자인 시스템 (현재 값)

- 색상: 배경 `#f3eee7`/`#f6f1e8` 베이지 계열, 잉크 `#161310`/`#201a15`, 뮤트 텍스트 `#6f675f`/`#7b6f64`, 포인트 컬러(accent) `#9b7a59`/`#a67c52` — 두 페이지가 변수명·값이 살짝 다름 (예: index는 `--ink`, portfolio는 `--text`).
- 라운드: `--radius-xl 34px / lg 26px(24px) / md 18px` 로 전체적으로 큰 pill·카드 라운드 사용.
- 그림자: `--shadow`로 부드러운 warm-brown 톤 큰 블러 그림자 통일.
- 레이아웃: `.wrap`/`.shell` 최대폭 1220~1240px, 카드형 섹션(`section-panel`, `step-shell`, `.section`)이 반복되는 "종이 카드 위에 카드" 구조.
- 반응형 브레이크포인트: 1100px(또는 1080px), 720px(또는 760px) 2단계만 존재. 태블릿 세부 대응은 약함.
- 두 페이지의 디자인 토큰이 완전히 통합되어 있지 않음 (공통 CSS 파일 없이 각 HTML에 중복 정의) → 리뉴얼 시 디자인 시스템을 별도 파일/토큰으로 분리할 필요.

## 5. index.html 섹션 구조

1. **topbar**: 로고 + 앵커 네비게이션(원가견적서/소장님/비용절감/주거·상업공간/공사과정/상담신청).
2. **hero**: 좌측 카피(9.9% 마진 문구, CTA 2개, 지표 3개, 이미지 갤러리 3장) + 우측 상담 신청 폼(성함/연락처/현장위치/예산/상담방식/참고사항 + 개인정보 동의).
3. **director (소장님 소개)**: 4명의 소장 프로필 카드(사진+역할+이름+경력) — 이미지가 `director-*-{color}.png`로 색상별 톤만 다르게 명명됨(실사진 아님, placeholder 성격).
4. **estimate (Step 2, 원가 견적표)**: 6행짜리 정적 테이블(구분/공사유형/설명/인건비원가/자재비원가/비고).
5. **saving (Step 3, 비용 절감 사례)**: 5행짜리 정적 테이블(사례명/공간유형/조정포인트/절감내용/절감금액) — 사례명이 "바디코드", "지니샌드", "코드페어" 등 의미 불명확한 placeholder 텍스트.
6. **cases (시공 사례 갤러리)**: 주거공간 9개 + 상업공간 9개, 각 카드가 `portfolio.html?slug=...` 로 딥링크.
7. **process (공정 흐름 + 추천 대상)**: 5단계 프로세스 로우 + 4개 추천 카드.
8. **footer**: 브랜드명(`interior connect` — 상단 브랜드명 "인테리어 인스턴트"와 불일치), 회사정보(전부 "추후 입력" placeholder), 제휴문의/원가견적 버튼(둘 다 `#contact`로 연결되나 실제 `#contact` 앵커 요소 없음 — 깨진 링크), SNS 링크(`#`만 있음).
9. **상담 폼 JS**: `submit` 이벤트에서 `preventDefault` 후 `alert`만 띄우고 끝 — 실제 전송/저장 로직 없음(README에 명시된 대로 데모용).

## 6. portfolio.html 구조 및 로직

- `?slug=` 쿼리파라미터로 `portfolioMap` 객체에서 항목을 찾아 JS로 전체 DOM을 그리는 SPA형 단일 페이지(서버 라우팅 없음). slug 없거나 매칭 실패 시 `renderEmptyState()` 렌더.
- `portfolioMap`: residential-1111~9999(9개), commercial-11~99(9개) 총 18개 항목. 각 항목에 category/label/title/subtitle/hero/area/duration/cost/scope/estimateKey/description/points[]/gallery[3] 하드코딩.
- **견적 자동 계산 로직** (`buildEstimate`):
  - `item.cost` 문자열("총 공사비 3,480만원")을 정규식으로 파싱해 총액 산출.
  - 마진 9.9% 역산: `costBeforeMargin = total / 1.099`, `margin = total - costBeforeMargin`.
  - `estimateTemplates.residential` / `.commercial`에 정의된 공정별 인건비·자재비 비중(weight)으로 항목별 금액 분배.
  - 즉, 실제 견적이 아니라 "총 공사비" 하나의 숫자를 비율로 쪼개서 그럴듯한 표로 보여주는 시뮬레이션.
- 하단 플로팅 바: 스크롤 상관없이 상시 노출, 현재 보고 있는 항목의 원가견적 총액 표시 + "현재 디자인으로 문의하기" 버튼.
- 문의 모달(`inquiryModal`): 성함/연락처(3분할 input)/현장이름·링크/내용/개인정보동의. 제출 시 유효성 체크 후 `successModal`만 띄우고 실제 전송 없음 (index.html 폼과 동일하게 데모).
- index.html과 마찬가지로 CSS 변수, 카드 스타일, 폰트가 별도로 재정의되어 있어 완전히 독립된 파일.

## 7. 데이터/콘텐츠 상 리스크 (리뉴얼 전 확인 필요)

- 회사 정보(사업자등록번호/주소/전화/이메일)가 전부 "추후 입력" 상태 — 실제 배포 전 필수 채움 항목.
- 브랜드명이 페이지마다 다름: 상단 `인테리어 인스턴트` / 브랜드마크 `IN` vs 푸터·portfolio topbar `interior connect` / `INTERIOR CONNECT`.
- 상담/문의 폼 2종 모두 실제 저장·전송 로직 없음 (README에 구글폼/이메일/카카오/CRM 연동 필요하다고 명시됨).
- 원가 견적 수치, 절감 사례명, 소장 4인, 시공 사례 18건 모두 실제 데이터가 아닌 예시 텍스트/숫자로 보임 — 실사진·실사례 교체 필요.
- 이미지 파일명이 `case-*`, `commercial-11~99`, `residential-1111~9999`처럼 규칙이 제각각이라 유지보수 시 헷갈리기 쉬움. `debug-alpha-crop.png` 같은 디버그성 파일도 남아있음.
- `#contact` 앵커가 실제로 존재하지 않아 footer의 "제휴문의"/"원가견적 신청하기" 링크가 아무 동작도 안 함 (실제로는 `#top`이나 상단으로 이동만 함).
- 두 파일 다 2000줄에 가까운 단일 HTML(CSS+JS 인라인)이라 재사용/유지보수성이 낮음 — 컴포넌트/템플릿 분리가 안 되어 있음.

## 8. "실제 개발물을 얹기 위한" 관점에서의 시사점

리뉴얼 목표가 "디자인·구성만 다시 잡아서, 추후 실제 백엔드/CMS 개발물을 바로 얹을 수 있게" 하는 것이므로, 현재 구조에서 짚어둘 지점:

- 데이터(포트폴리오 항목, 견적 항목, 소장 프로필, 회사 정보)가 전부 HTML/JS에 인라인되어 있음 → 추후 API/CMS 연동을 고려한다면 이 데이터들을 별도 JSON/데이터 레이어로 분리해두는 편이 이식이 쉬움.
- 상담 폼 2종은 지금은 `alert`로 끝나지만, 구조상 `<form>` + `submit` 이벤트 훅이 이미 잡혀 있어 나중에 API 엔드포인트만 연결하면 되는 형태.
- `portfolio.html`은 쿼리파라미터 기반 클라이언트 렌더링 구조라, 추후 실제 라우팅(Next.js 등 프레임워크 도입 시) 이전 시 `portfolioMap`을 그대로 데이터 소스로 재사용 가능.
- CSS가 두 파일에 중복 정의되어 있어, 리뉴얼 시 공통 디자인 토큰(색상/라운드/그림자/타이포)을 먼저 정리하고 시작하면 이후 컴포넌트 이식이 수월함.
- 현재 4개 섹션(소장님 소개/원가견적표/비용절감사례/시공사례) 자체는 참고 사이트의 핵심 기능 흐름을 그대로 따온 뼈대이므로, 리뉴얼 시에도 이 정보 구조(IA) 자체는 유지하되 비주얼과 실데이터만 교체하는 방향이 안전해 보임.
